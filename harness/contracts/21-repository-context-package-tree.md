## 0. 2026-06-12 协作修订口径：产品默认是共享 authored harness

Harness Anything 的产品默认不是“每个协作者一个 private harness repo”。默认模型是：

1. **共享 authored root** 跟随目标代码仓库进入 Git，承载团队共同编辑的任务、标准、模板和决策。
2. **本地 generated/runtime root** 默认 gitignored，承载可重建投影、cache、journal、locks 和 adapter snapshots。
3. **private harness root** 只用于私有部署、bootstrap 本仓重写计划或不能公开的运营资料，不是开源产品默认布局。

这不是单纯目录命名问题，而是多人 PR 的 merge contract。后续 `harness init`、`new-task`、`check`、
`governance rebuild` 必须按本节解释实现。

      relation-graph.json
      lifecycle-status.json
      closeout-readiness.json
      dashboard-bundle.json

Product `.gitignore` generated by `harness init` must ignore `.harness/` by default. If a team chooses a custom
authored root, the same split still applies: authored root tracked, generated/runtime root ignored.

## 2. Task package skeleton

```text
<task>/
  INDEX.md                 # identity, lifecycle binding, packageDisposition, nav
  task_plan.md             # goal boundary, scope, dependencies, stop rules
  findings.md              # facts/risks/decisions discovered while working
  progress.md              # append-oriented progress log and evidence pointers
  references/INDEX.md      # source materials, external docs, uploaded files
  artifacts/INDEX.md       # generated outputs, patches, reports, screenshots
  walkthrough.md           # closeout narrative; not worker startup default
  lesson_candidates.md     # reusable lessons; not worker startup default
  review.md                # vertical-specific review packet when enabled
```

Worker default read set：

1. `INDEX.md`
2. `task_plan.md`
3. `findings.md`
4. latest bounded tail of `progress.md`

Closeout/review read set：adds `walkthrough.md`、`lesson_candidates.md`、`review.md`、artifacts index。

## 4. Context package rules

- Authored facts live in task package docs。
- Generated views live under `.harness/generated/` by default and can be deleted/rebuilt。
- External snapshots live under `.harness/cache/snapshots/`, never in authored truth。
- Private-only evidence can live under task package but must be marked `visibility: private-only`。
- Public release docs must only consume `PublishableProjection`。
- Explicit generated exports are allowed only when a human/product workflow needs a durable snapshot. Exported files
  must live under authored `artifacts/` or a named release artifact path, carry provenance (`sourceHash`,
  `generatedAt`, `generatorVersion`), and never become the source of truth.

## 7. Hidden operational state

`.harness/` is implementation-owned, not authored knowledge：

- `cache/projections.sqlite`: rebuildable read model。
- `cache/snapshots`: external last-known snapshots, non-authoritative。
- `write-journal`: durable op queue, authoritative only until flushed into authored docs。
- `locks`: advisory locks; no semantic meaning。

Do not ask agents to edit `.harness/` manually。

## 8. Merge contract for shared authored harness

Git merge is the cross-person concurrency boundary. Local lock files protect only processes in one clone; they do not
prevent two people from creating conflicting authored changes in separate PRs.

### 8.1 Authored file merge constraints

- Task identity is a random stable `task_<ULID>` in frontmatter. Directory slug is only a locator/display hint.
- New task packages are union-merge friendly: each task lives in its own directory, progress is append-oriented, and
  generated aggregate tables are not edited by hand.
- Shared standards must be split into narrow files under `harness/standards/*.md`. `AGENTS.md` stays a short shim and
  should not accumulate volatile instructions or generated status.
- Coordinator-owned global ledgers are allowed, but they must have one owner path and a checker rule. Subagents do not
  opportunistically edit shared generated/status tables.

### 8.2 Post-merge checker

`harness check --post-merge [--json]` is the required product surface after pulling or merging another branch. It must
detect at least:

| Invariant | Severity | Code | Agent guidance |
| --- | --- | --- | --- |
| duplicate `task_id` across authored packages | hard-fail | `duplicate_task_id` | keep both packages only after assigning a new random ID to the accidental duplicate and updating internal refs |
| duplicate `(engine, ref)` binding across authored packages | hard-fail | `duplicate_external_binding` | keep one binding; archive or supersede the accidental duplicate because binding blocks are immutable |
| generated/runtime files tracked by Git | hard-fail | `generated_tracked` | remove from Git index, keep under `.harness/`, regenerate if needed |
| binding block tampered after creation | hard-fail | `binding_tampered` | do not edit binding; archive + create replacement task and relate via `supersedes` |
| conflict markers in authored harness files or root AGENTS/CLAUDE shims | hard-fail | `conflict_marker_present` | resolve markers in authored docs, then run `harness check --post-merge` again |
| authored references point to missing tasks | hard-fail | `dangling_entity_ref` | restore target package, update ref, or record a supersedes/deleted relation |
| authored parent-of / depends-on relation cycle after merge | hard-fail | `relation_cycle_detected` | remove or split the relation that closed the cycle; rerun `harness check --post-merge` |
| directory slug conflicts for different task IDs | warning | `task_slug_conflict` | keep IDs stable; rename locator directory or add disambiguating suffix |
| generated projection stale or missing | warning | `projection_stale` | run `harness governance rebuild` |
| AGENTS.md oversized or carrying volatile status | warning | `agent_entry_scope_violation` | move durable rules into split `harness/standards/*.md`; keep shim one-level |
| standards file has ambiguous ownership | warning | `standards_owner_ambiguous` | assign owner/CODEOWNERS entry or split the standard into narrower owned files |

Human-readable output must include a short prompt block that an agent can follow. JSON output must preserve stable error
codes, affected paths, and suggested next command.

### 8.3 CI gate

Public repos using Harness Anything should run the same post-merge invariants in CI. CI may skip local-only freshness
checks that require external credentials, but it must fail on authored identity, generated-tracking, binding, and conflict
marker violations.
