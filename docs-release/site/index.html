Harness Anything

Try it · about 10 minutes上手 · 约 10 分钟

Your agent did the work. Where did the reasoning go? 代理把活干完了。可那些推理去哪了？

An AI agent fixes a login bug at 2 a.m. It weighs two approaches, picks one, notes what it observed — then the session ends and all of that lives in a chat transcript nobody will ever reopen. Harness Anything is a small command-line tool, ha, that captures that trail as durable records in your git repo instead: things you can review in a normal diff, query, and build on. 凌晨两点，一个 AI 代理修好了一个登录 bug。它权衡了两种方案、选了一种、记下沿途的观察——然后会话一结束，这一切就躺进了一份没人会再打开的聊天记录里。Harness Anything 是一个小小的命令行工具 ha，它换一种做法：把这条轨迹变成你 git 仓库里可长期留存的记录——能在普通 diff 里审查、能查询、能在其上继续构建。

Run one command and structure starts accumulating in your repo. Below is a real loop, captured from an actual run — no screenshots, just the tool. 运行一条命令，结构就开始在你的仓库里积累。下面是一个真实的循环，取自一次实际运行——没有截图，就是这个工具本身。

01

What it is, in one screen一屏说清它是什么

Harness Anything turns the decisions, tasks, and facts your agents produce into first-class records on git — queryable, reversible, and reusable — instead of losing them inside a transcript. Every one lands as plain Markdown you can read in a git diff. Harness Anything 把你的代理产生的决策（decision）、任务（task）和事实（fact）变成 git 上的一等公民记录——可查询、可回滚、可重用——而不是把它们丢失在聊天记录里。每一条都落地为纯 Markdown，你可以在 git diff 里读它。

Three things to take away: 三件要点：

  • It solves the "where did the reasoning go?" problem — agent work stops evaporating into logs.它解决了"推理去哪了？"的问题——代理的工作不再蒸发进日志。
  • Unlike note-taking, these are structured, linked records with a lifecycle: decisions can be overturned; facts are anchored to the task that observed them.和记笔记不同，这些是结构化、有链接、带生命周期的记录：决策可以被推翻，事实锚定到观察到它们的那个任务。
  • You need Node.js 24+ and git. There's no public npm release yet — you install it globally from source (npm install -g .) so the ha command is on your PATH.你需要 Node.js 24+ 和 git。目前还没有公开的 npm 发布——你从源码全局安装（npm install -g .），让 ha 命令进入你的 PATH。

Why · the ideas为什么 · 背后的想法

AI agents are great at doing work — and terrible at remembering why. AI 代理很擅长做事——却极不擅长记住为什么。

Give an agent a real task and it generates, in one session, a dense trail: why it chose this over that, how far it got, what it saw. Then the session ends and almost all of it evaporates. This tab is the argument for a fix — not better notes, but turning that trail into structure that lives in git. 给代理一个真实任务，它会在单次会话里生成一条密集的轨迹：为什么选这个而不是那个、做到了哪一步、看到了什么。然后会话结束，几乎全部蒸发。这个标签页讲的是一个解法——不是"记更好的笔记"，而是把那条轨迹变成活在 git 里的结构。

01

The entropy problem熵问题

Every session produces three kinds of durable content, whether or not anyone captures it — choices ("we're going with Postgres, not SQLite, because…"), progress ("the migration is half done; the index rebuild is blocked"), and observations ("at 10k QPS the p99 latency was 50ms"). 每个会话都会产生三类持久内容，不管有没有人去捕捉它——选择（"我们选 Postgres 而不是 SQLite，因为……"）、进度（"迁移做到一半；索引重建卡住了"）、观察（"10k QPS 下 p99 延迟是 50ms"）。

By default these live in the chat transcript. So once the conversation scrolls away, the why is gone, the state is gone, and the evidence is gone. The next agent — or the next you, a week later — starts cold. Decisions get silently re-litigated. Work gets redone. Nothing is auditable, because there's no artifact to audit. 默认情况下，这些都活在聊天记录里。于是对话一翻篇，"为什么"没了，"进度"没了，"证据"也没了。下一个代理——或者一周后的你自己——只能从零开始。决策被悄悄地重新讨论一遍。工作被重做。整件事都无法审计，因为根本没有可审计的工件。

This is entropy: information produced but never consumed, accumulating until it's indistinguishable from noise. A pond that only takes inflow eventually turns green. 这就是熵：产生了却从未被消费的信息，不断堆积，直到和噪声再也分不清。一个只进水不出水的池塘，迟早会变绿发臭。

02

The bet: make these first-class, in git我们的赌注：把它们做成一等公民，存进 git

The fix isn't better note-taking. It's promoting that trail into structured entities that live in git — Markdown as the source of truth, the repository as the single place the truth lives. 解法不是"做更好的笔记"，而是把那条轨迹提升为活在 git 里的结构化实体——以 Markdown 作为真相的载体，代码库作为真相唯一的栖身之所。

Not a scratchpad — entities with lifecycles, state machines, and typed relations between them. A choice becomes a decision that can be proposed, accepted, and later overturned, with a paper trail. Progress becomes a task that moves through defined states. An observation becomes a fact that is immutable and carries its own provenance. Because they're plain files under version control, they're diffable, reviewable, and permanent, and any agent can pick them up without a warm conversation to inherit. 这不是一块随手记的草稿板，而是一组有生命周期、有状态机、彼此之间存在类型化关系的实体。选择变成一个可被提议、被接受、日后也可被推翻的决策，并留下完整记录。进度变成一个在既定状态间流转的任务。观察变成一个不可变、自带来源的事实。因为它们只是版本控制下的纯文本文件，所以可对比差异、可审查、能长期留存，任何代理接手时都不需要先补一段热身对话。

03

The kernel is only three primitives内核只有三个原语

Everything else is a combination of these three. Here's each one, in plain words, on first use: 其他一切都是这三者的组合。先把每一个用大白话讲清楚：

decision — the WHY—— 为什么
a why frozen into a commitment you can later reverse. It has alternatives and evidence, and it moves proposed → accepted → active → retired / rejected / deferred.一个"为什么"被冻结成一个你日后可以推翻的承诺。它带着备选项和证据，并沿 proposed → accepted → active → retired / rejected / deferred 流转。
task — the WHAT—— 做什么
a what in motion, with real states — planned, active, blocked, in‑review, done, cancelled — and operations to move between them.一个"做什么"在流转中，拥有真实的状态——planned、active、blocked、in‑review、done、cancelled——以及在这些状态之间移动的操作。
fact — the IS—— 是什么
an is: born immutable, it never changes. If the world changes, you don't edit the old fact — you record a new one.一个"是"：生来不可变，永远不会再改。如果现实变了，你不去改旧的事实，而是记录一个新的。

They interlock into a closed loop. A decision derives a task; executing the task produces facts; those facts are referenced back by decisions as evidence. Remove any one and the loop can't close. 它们咬合成一个闭环。一个决策衍生出一个任务；执行任务产生事实；这些事实又被决策反向引用作为证据。拿掉任何一个，环就闭不上。

They even store asymmetrically. Decisions are centralized — the spine, the one projection a human is meant to watch. Tasks are containers, each its own directory. Facts are embedded in the task that produced them — because a fact without its task is untrustworthy. Take "redis p99 = 50ms at 10k QPS" out of context and you've stripped its provenance. The rule: don't move it, reference it. 它们的存储方式甚至是不对称的。决策是集中存放的——主干，人类该盯着看的那个唯一投影。任务是容器，各自一个目录。事实是嵌入在产生它的任务里的——因为脱离了任务的事实是不可信的。把"redis 在 10k QPS 下 p99 = 50ms"抽离语境，你就剥掉了它的出处。口号是：不要搬它，引用它。

04

Decision vs. verdict — the one confusion that matters决策 vs 裁决——最要紧的一个混淆

A decision answers which path do we take? — a load-bearing choice you can later supersede. A verdict answers does this one output hold? — a one-shot PASS/FAIL judgment on a concrete result, produced when a task goes through review or completion. A decision is a cause; a verdict is an effect. 决策（decision）回答的是我们走哪条路？——一次承重的选择，你日后可以推翻（supersede）。裁决（verdict）回答的是这个输出成立吗？——在任务走 review 或 completion 时，对某个具体结果的一次性 PASS/FAIL 判断。决策是原因；裁决是结果。

Why keep them apart? Because if every routine PASS/FAIL is funneled through the decision machinery, the decision queue fills with per-output bookkeeping until no one can see it. Routine verdicts are the flood; keeping them out is what keeps the decision queue meaningful. A verdict does not get a dec_ id, does not enter the decisions directory, and does not go on the queue. 为什么要分开？因为如果每一次日常的 PASS/FAIL 都被塞进决策这套机制里，决策队列就会被大量逐条记账式的检查结果淹没，直到没人看得清它。日常的裁决就是那场洪水；把它们挡在队列之外，才能让决策队列继续有意义。裁决拿不到 dec_ 式的 id，不进入 decisions 目录，也不上队列。

And the routing is not automatic. Nothing turns a FAIL into a new decision on its own. Only when a verdict surfaces something strategic ("these results suggest we chose the wrong path") does it prompt a human to propose a new decision — a deliberate act. 而且这条路由不是自动的。没有任何东西会自己把一个 FAIL 变成一个新决策。只有当裁决暴露出某种战略性问题时（"这批结果说明我们可能选错了路"），它才会促使人去提出一个新决策——一个刻意的动作。

05

Every load-bearing write passes a gate — and gates fail closed每个承重写入都要过一扇门——而门默认关闭

The core rule: every load-bearing write passes through a gate, and the gate is closed by default. If the evidence is insufficient or a check doesn't pass, the write is rejected, not waved through. The safe default is "no"; you have to earn the "yes". 核心规则：每一次承重写入都必须经过一扇门，而门默认是关闭的。证据不足，或某项检查没通过，写入就会被拒绝，而不是被放行。安全的默认值是"否"；你必须自己赚得那个"是"。

A fail-open system trusts that nothing went wrong. Fail-closed assumes something might have, and makes you prove otherwise. Load-bearing writes never touch disk directly — they go through a single write coordinator that stamps and commits every accepted write, so there's one place to enforce the rule and every accepted write leaves a durable, attributable trace. 一个默认开放的系统信任"什么都没出错"。fail-closed 则假设"可能出错了"，要求你先证明没出错。承重写入从不直接落盘——它们经过唯一的写协调器（write coordinator），它给每次被接受的写入盖章并提交，于是只有一个地方执行规则，而每次被接受的写入都留下持久、可追溯的痕迹。

One gate is a hard floor: a task cannot enter in_review or done unless its facts.md holds at least one real fact. "Done" that observed nothing isn't done. 有一道门是硬底线：一个任务无法进入 in_review 或 done，除非它的 facts.md 里至少有一条真实的事实。什么都没观察到的"完成"，不算完成。

06

The adoption law — and four layers of "done"采用律——以及"完成"的四层

A capability is adopted if and only if it is the sole legal path on a load-bearing route. If a legacy alternative still exists and migration isn't forced, the legacy wins by inertia. The mechanism has to be the road, not a sign beside it. 一项能力被采用，当且仅当它是某条承重路径上唯一的合法路径。只要遗留方案仍然存在、迁移没有被强制，遗留方案就会凭惯性获胜。这套机制必须就是那条路，而不是立在路边的一块提示牌。

That reframes what "done" means. It has four layers: 这重新定义了"完成"。它有四层：

  1. Structural justice — tests & CI green, no hard-gate violations. Proves the code runs, not that it matters.结构正义——测试和 CI 全绿，没有硬门违规。证明代码能跑，证明不了它有意义。
  2. Semantic acceptance — the implementation matches the decision it claims to implement. Not a plausible neighbor; the thing itself.语义验收——实现真正匹配它所声称要落实的那个决策。不是相似物，而是那个决策本身。
  3. Adversarial verification — the load-bearing pieces survive a double-blind red team.对抗式验证——承重的部分扛得住双盲红队测试。
  4. Usage proof — real usage appears after delivery, or the thing is the forced path with no legacy alternative left.使用证明——交付后真实使用出现，或者这东西本身就是强制路径、不再有遗留替代方案。

Something that clears the first three but fails the fourth is shipped-unused: it exists, it works, nobody reaches for it. That does not count as a completed milestone. The reinforcing signal is the mirror image: the structures we unconsciously reach for by hand, over and over, are the innovations already validated by behavior. 通过了前三层却在第四层失手的东西，叫作已发布未使用（shipped-unused）：它存在，能运行，但没人用它。这不算完成的里程碑。而强化信号是它的镜像：我们无意识地反复用手工去够的那些结构，正是已经被行为验证过的创新。

07

Where it's going它往哪走

A note on direction, not a schedule — no dates, no ordering, all of it subject to change. The base is settled: the three-primitive kernel and a working CLI, already used to manage its own development. Everything below grows outward from that. 这是一段关于方向的说明，不是时间表——没有日期，没有先后，一切都可能变。基础已经定下：三原语内核加一个能用的 CLI，并且已经在用来管理自身的开发。下面的一切都从这里向外生长。

  • Self-hosting the loop — turn the tool fully on its own development. It's the only honest proof of usefulness: a tool that can't carry its own work has no business carrying anyone else's.让循环自我托管——把工具彻底用在它自己的开发上。这是有用性唯一诚实的证明：连自己的工作都扛不起来的工具，没资格去扛别人的。
  • A visual surface — let people who don't live on the command line still see the structure, since the command line is the real adoption bottleneck and most people meet a tool through its surface.一个可视界面——让不常泡命令行的人也能看到结构，因为命令行才是真正的采用瓶颈，而大多数人是通过表层认识一个工具的。
  • New scenarios — bring domains beyond coding (research, operations) onto the same vertical model. It's the test of the central bet: a kernel that knows nothing about any domain should absorb a new one purely through a declaration.新场景——把编码之外的领域（研究、运维）纳入同一套场景模型。这是对核心赌注的检验：一个对任何领域一无所知的内核，应该仅凭一份声明就能吸收一个新领域。

One thing is deliberately not on the list: widening the kernel. New surfaces and new domains arrive in the layers around it; the moment a direction seems to need a new primitive, that's a signal to reexamine the direction, not the kernel. 有一件事刻意不在清单上：拓宽内核。新的界面和领域都出现在它周围的层里；一旦某个方向看起来需要一个新原语，那是重新审视方向的信号，而不是审视内核。

01

Two stores, and they are not peers两个存储，而它们并不对等

Markdown in your git repository is the source of truth. SQLite is a rebuildable projection — a fast read cache you can delete and regenerate from the Markdown at any time. 你 git 仓库里的 Markdown 是真相的来源。SQLite 是一个可重建的投影——一个快速的读缓存，你随时可以删掉它，再从 Markdown 重新生成。

Every entity — decision, task, fact — is a plain Markdown file with YAML frontmatter, committed to git. Nothing about correctness depends on the database surviving: delete it and the next read rebuilds it from the files. When the two disagree, the Markdown wins by definition, and a freshness check exists precisely to notice. Hold onto that asymmetry — it explains the whole layout: writes flow down toward Markdown and git; reads are served from a projection any write can invalidate and any command can rebuild. 每个实体——decision、task、fact——都是一份带 YAML frontmatter 的纯 Markdown 文件，提交进 git。正确性不依赖数据库的存活：删掉它，下一次读取就从文件把它重建出来。当两者不一致时，按定义 Markdown 说了算，而一个新鲜度检查的存在正是为了发现这种不一致。记住这处不对称——它解释了整个布局：写入朝着 Markdown 与 git 向下流动；读取则由一个任何写入都能让它失效、任何命令都能把它重建的投影来提供。

02

How the three entities live on disk三个实体在磁盘上怎么住

Every entity is the same physical thing: a Markdown file with a YAML frontmatter block on top. The frontmatter isn't decoration — it's validated against a schema in packages/kernel/src/schemas/ before the file is accepted, so the fields are contracts, not conventions. Three primitives, but only two storage sites: decisions centralized, tasks as containers, facts embedded. 每个实体都是同一种物理东西：一份顶部带 YAML frontmatter 块的 Markdown 文件。frontmatter 不是装饰——在文件被接受之前，它要经 packages/kernel/src/schemas/ 里的 schema 校验，所以这些字段是合同，不是约定。三个原语，却只有两个存储点：决策集中，任务作容器，事实嵌入。

The IDs are shaped and enforced. A decision id matches dec_…. A task id is task_<ULID>-<slug> — the ULID is sortable and unique, the slug is human-readable. A fact id is F- plus exactly eight Crockford base32 characters (digits and uppercase, excluding I L O U), e.g. F-3DQ0KZRV. 这些 ID 有固定形状并被强制。决策 id 匹配 dec_…。任务 id 是 task_<ULID>-<slug>——ULID 可排序且唯一，slug 人类可读。事实 id 是 F- 加上恰好八个 Crockford base32 字符（数字和大写字母，排除 I L O U），例如 F-3DQ0KZRV。

The decision file's frontmatter is validated against decision-package/v1 and carries load-bearing fields like question, chosen[], rejected[] (each with a why_not), claims[], relations[], and provenance[]. One rule is enforced by the schema itself: a decision whose proposedBy equals its arbiter is rejected — you cannot arbitrate your own proposal, and a record of that shape never reaches disk. 决策文件的 frontmatter 依 decision-package/v1 校验，携带承重字段，如 question、chosen[]、rejected[]（每个带一个 why_not）、claims[]、relations[] 和 provenance[]。有一条规则由 schema 本身强制：proposedBy 等于 arbiter 的决策会被拒绝——你不能裁决自己的提议，这种形状的记录根本到不了磁盘。

Each of the three primitives has its own frontmatter schema and each is versioned in the name itself: a task's INDEX.md declares task-package/v2, a decision declares decision-package/v1, and a fact line is validated against fact-record/v1. The version suffix is a contract, not a label — the loader knows exactly which shape to expect, and a file that doesn't match its declared schema is refused at the door rather than half-read. 三个原语各有自己的 frontmatter schema，而且版本号就写在名字里：任务的 INDEX.md 声明 task-package/v2，决策声明 decision-package/v1，事实行则依 fact-record/v1 校验。版本后缀是合同，不是标签——加载器清楚知道该期待哪种形状，而一份对不上自己所声明 schema 的文件会在门口被拒，而不是被读到一半。

03

The single write path — the seal唯一的写路径——那枚印章

Load-bearing writes — creating a task, transitioning it, appending progress, proposing or accepting a decision — never touch disk directly. They go through one component, the write coordinator (WriteCoordinator in packages/kernel/src/ports/write-coordinator.ts), whose interface is tiny on purpose: 承重写入——创建任务、迁移它、追加进度、提议或接受决策——从不直接落盘。它们经过一个组件，写协调器（packages/kernel/src/ports/write-coordinator.ts 里的 WriteCoordinator），它的接口刻意做得很小：

Every write splits into two phases so a crash leaves a recoverable state, not a corrupt one. Enqueue records intent: it validates the op, runs a preflight check, and appends a record to an append-only journal — nothing in your tree has changed yet. Flush produces the effect: under a repository lock it applies each pending op to disk (atomically, via a temp-file write then rename), commits the touched paths to git, and finally writes a watermark — the authoritative record of what has been committed. 每次写入分成两个阶段，好让崩溃留下一个可恢复的状态，而不是一个损坏的状态。Enqueue 记录意图：它校验 op、跑一次预检、往一个只增不改的 journal 追加一条记录——你的工作树此刻还什么都没变。Flush 产生效果：在仓库锁下，它把每条待处理的 op 原子地应用到磁盘（先写临时文件再改名），把被触碰的路径提交到 git，最后写一个水印（watermark）——已提交内容的权威记录。

That watermark is the wax seal. Each decision document also gets a _coordinatorWatermark stamped into its frontmatter; its presence proves the file went through the single write path, and post-merge checks hard-fail if it's missing or duplicated. The git commit carries a semantic message like task(transition): <id> → in_review [<opId>]. There is no "write these bytes to that path" primitive — if an operation isn't one of the enumerated kinds, it cannot enter the system. 那个水印就是蜡封。每个决策文档还会被在 frontmatter 里盖上一个 _coordinatorWatermark；它的存在证明这份文件走过了那条唯一的写路径，合并后的检查会在它缺失或重复时硬失败。git 提交带一条语义信息，如 task(transition): <id> → in_review [<opId>]。系统里没有"把这些字节写到那个路径"的原语——如果一个操作不属于被枚举的种类之一，它就进不来。

Read that as one design move: stamp, then commit — one door. Because every load-bearing write ends by stamping a watermark and then committing to git, there is exactly one place where structure becomes durable, and exactly one artifact to audit afterward. A record without its watermark, or one that never became a git commit, simply isn't a real write — and the checks that run after a merge are built to notice precisely that. 把这理解成一个设计动作：先盖章，再提交——一扇门。因为每一次承重写入都以盖一个水印、再提交到 git 收尾，所以结构变得持久的地方只有一个，事后可审计的工件也只有一个。一条没有水印、或从未变成 git 提交的记录，根本不算一次真实的写入——而合并之后跑的检查，正是为发现这一点而造的。

04

The projection: Markdown to SQLite投影：从 Markdown 到 SQLite

The read side is served from a SQLite file at .harness/cache/projections.sqlite — a rebuildable projection, safe to delete because the Markdown in git is authoritative and the rebuild is a pure, reproducible function of the files. Rebuild is a single pass: scan the Markdown (task INDEX.md files, decision docs), validate and convert each entry into a typed row, compute content hashes, build the relation graph, then write a fresh SQLite to a temp file and rename it into place atomically. 读取一侧由一个 SQLite 文件提供，位于 .harness/cache/projections.sqlite——一个可重建的投影，删掉也安全，因为 git 里的 Markdown 是权威，而重建是这些文件的一个纯粹、可复现的函数。重建是单趟：扫描 Markdown（任务 INDEX.md 文件、决策文档），校验并把每条 entry 转成类型化的行，计算内容哈希，构建关系图，然后把全新的 SQLite 写到临时文件并原子改名到位。

Freshness is checked on every read. If the stored sourceHash matches the Markdown, the read is served from the database. If it differs, the projection is stale and gets rebuilt transparently. If a stored row-hash doesn't match — a sign of tampering — it hard-fails and rebuilds. Missing or unreadable file? Rebuilt. The cache holds real tables (task_projection, decision_projection, relation_edges, relation_coverage, task_fact_anchors, and a projection_meta row of fingerprints), but none of them is a source of truth. 每次读取都会检查新鲜度。如果存下的 sourceHash 与 Markdown 匹配，读取就从数据库来。如果不同，投影就是陈旧的，会被透明重建。如果某个存下的行哈希对不上——篡改的迹象——它会硬失败并重建。文件缺失或读不出？重建。这个缓存里有真实的表（task_projection、decision_projection、relation_edges、relation_coverage、task_fact_anchors，还有一行指纹用的 projection_meta），但它们没有一个是真相的来源。

The practical upshot: this whole SQLite file is disposable. It lives under .harness/, stays out of git, and can be deleted at any moment with zero loss — the next command rebuilds it from the Markdown that git already holds. That's what lets the tables above be aggressive and denormalized for fast reads without ever becoming a liability: a projection you can always throw away is one you never have to migrate, back up, or trust over the files. 实际的结论是：整个 SQLite 文件是可丢弃的。它住在 .harness/ 下，留在 git 之外，随时可以删掉而毫无损失——下一条命令会从 git 早已保存的 Markdown 把它重建出来。正因如此，上面那些表可以为了读取速度而激进地反范式，却永远不会变成负担：一个你随时能扔掉的投影，就是一个你永远不必迁移、不必备份、也不必凌驾于文件之上去信任的投影。

06

Verticals: the declaration engine场景合同：声明式引擎

The kernel is the part deliberately kept small — decisions, tasks, facts, relations. New domains don't arrive by rewriting it; they arrive as declarative verticals. A vertical is a single vertical.json, validated against VerticalDefinitionSchema, that declares the entity kinds, document scaffolds, checker rules, and allowed relation types for a domain. The engine reads the declaration; the kernel that reads it does not change. 内核是被刻意保持精简的那部分——decision、task、fact、关系。新领域不是靠重写它来加入的；它们作为声明式的场景合同（vertical）加入。一个场景合同是单个 vertical.json，经 VerticalDefinitionSchema 校验，声明某个领域的实体种类、文档脚手架、检查器规则和允许的关系类型。引擎读这份声明；读它的那个内核不变。

Entity kinds come in flavors: a lifecycle kind (like task or decision) declares a packageKind such as task-package/v2 and gets a full document package; a schema kind (like fact) declares a schemaRef and constrains fields only, with no document template. The guiding rule is convention over declaration: if the engine can infer something from files that already exist — directory structure, file names, schema validity — that's convention; only what no filesystem can reveal (a load-bearing flag, a checker profile, a locale fallback) is declared, and kept minimal. 实体种类分几种：一个生命周期种类（如 task 或 decision）声明一个 packageKind，比如 task-package/v2，并得到完整的文档包；一个 schema 种类（如 fact）声明一个 schemaRef，只约束字段，没有文档模板。指导原则是约定优于声明：如果引擎能从已经存在的文件里推断出某件事——目录结构、文件名、schema 合法性——那就是约定；只有文件系统无法揭示的东西（一个承重标志、一个检查器档案、一个语言回退）才被声明，而且尽量最小化。

A milestone is a coding idea; a research project might have phases instead. New surfaces and new domains arrive in the layers around the kernel — verticals, presets, tooling — while the primitives stay fixed. 里程碑是一个编码领域的概念；一个研究项目可能改用阶段（phases）。新的界面和新的领域出现在内核周围的那些层——场景合同、预设、工具——而原语本身保持不变。

07

Provenance, verdicts, and the event ledger出处、裁决与事件账本

Every entity carries a provenance[] array with at least one entry, so no record floats free of an origin. Each entry (ProvenanceEntrySchema) is exactly three non-blank fields: runtime (one of human, claude-code, codex, zcode, antigravity), sessionId, and boundAt. An entity can accumulate more than one binding as it passes through more than one run, but never zero. 每个实体都携带一个 provenance[] 数组，至少有一条，于是没有任何记录会脱离一个来源而漂浮。每条（ProvenanceEntrySchema）恰好是三个非空字段：runtime（human、claude-code、codex、zcode、antigravity 之一）、sessionId 和 boundAt。一个实体在经过多次运行时可以累积不止一条绑定，但永远不会是零条。

This is the machinery that keeps decision and verdict apart. A verdict is a PASS/FAIL judgment on one concrete output, recorded next to the work it judged — it gets no dec_-style id, never enters the decisions directory, and never goes on the decision queue. Turning a strategically significant verdict into a new decision is a deliberate human act, never automatic. 这正是让决策和裁决保持分开的机制。裁决是对某个具体输出的一次 PASS/FAIL 判断，被记录在它所判断的那项工作旁边——它拿不到 dec_ 式的 id，从不进入 decisions 目录，也从不上决策队列。把一个有战略意义的裁决变成一个新决策，是人刻意的动作，从不自动发生。

Underneath sits the event ledger — an append-only record of what happened during a run (runtime-event/v1). Each event has an evt_ id and a kind from a fixed set: session · turn · step · tool · approval · interrupt · result · cost. Its CLI surface has exactly two operations — append and list. No edit, no delete; the ledger only grows. That ledger is what the Exit Gate reads to confirm a piece of work is actually complete. 在这之下坐着事件账本——一次运行期间发生过什么的只增记录（runtime-event/v1）。每条事件有一个 evt_ id 和一个来自固定集合的 kind：session · turn · step · tool · approval · interrupt · result · cost。它的 CLI 面恰好两个操作——append 和 list。无编辑，无删除；账本只会增长。正是这个账本，被 Exit Gate 读取，用来确认一项工作确实完成了。
