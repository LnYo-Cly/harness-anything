# 路线图：方向，不是承诺

这是一份关于*方向*和*为什么*的陈述——不是*何时*或*如何*。它描述了我们认为值得追求的几个主题，以及它们为什么重要。它不承诺日期、机制或先后顺序，下面列出的顺序也不暗示任何次序。其中任何一项都可能变化。

## 现状

基础已经不只是一个扁平的三原语循环。源码 CLI 已经有 task、decision、fact、任务层级、关系语义，以及支持单机多仓的本地 daemon 路径。精确发布边界会随证据变化，所以状态真相源是
[发布状态边界](../../release-posture.zh-CN.md)。下面写的是方向；当前标签以那一页为准。

## 值得追求的方向

**让循环自我托管。** 把这套工具彻底用在它自己身上，让系统从头到尾管理自己的开发。这很重要，因为自我托管是有用性的唯一诚实证明：如果这个工具连自己的工作都扛不起来，就没资格去扛别人的工作。接下来真正有用的是在普通工作流里证明已经 ship 和 mechanism-complete 的命令面，而不是扩张内核。

**一个图形化的可视界面。** 降低上手门槛，让不常用命令行的人也能*看到*结构——决策、任务、层级、关系如何铺展开。这很重要，因为命令行才是真正的采用瓶颈；系统提供的大量东西在能被“看见”之前都是隐形的，而大多数人是通过一个工具的表层去认识它的，而不是它的内部。发布边界会刻意保守：一个能读取数据的源码 GUI，不等于完整桌面产品。

**一个不假装 remote-first 的本地 daemon。** 先让 daemon 稳定服务本地写协调和单机多仓，再去声称团队远程协作。这很重要，因为 daemon 是写入排序、投影和 GUI 读取路径交汇的地方；如果夸大 transport 形态，采用叙事反而会失去可信度。

**扩展到新场景。** 把编码之外的领域——研究、运维等等——纳入同一套场景模型，同时不改动内核。这很重要，因为这正是对核心赌注的检验：一个真正对任何领域一无所知的内核，应该仅凭一份场景声明就能吸收一个新领域。每一个契合的新场景都是“边界确实存在”的证据；第一个不契合的场景，则是我们能学到东西的地方。

## 我们刻意不去追求的

我们不会为了容纳上述任何一项而拓宽内核。整个前提就是：新的界面和新的领域都出现在内核*周围*的那些层——场景、预设、工具——而原语本身保持不变。一旦某个方向看起来需要一个新的原语，那就是一个信号：该重新审视的是这个方向，而不是内核。
